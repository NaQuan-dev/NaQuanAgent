# COMMON_ERRORS

本文件是可复用 Agent 框架的常见问题记录模板。这里只记录通用工程和安全经验，不记录任何组织、人员、客户、会话、密钥或业务系统细节。

## 使用原则

- 开始相关任务前，先按任务类型查看本文件。
- 遇到已记录的问题，优先执行对应策略并验证。
- 新增条目时只写通用现象、原因、策略和验证方式。
- 不写真实路径、真实 ID、真实 token、真实员工或客户名称。

## 通用问题：UTF-8 BOM 导致严格解析失败

场景：
- 使用旧版 shell 或编辑器写入 JSON、TOML、YAML 等配置文件。

典型现象/报错：
- 严格解析器提示文件开头有非法字符。
- 文件肉眼看起来正常，但程序无法读取。

原因判断：
- 写入工具在 UTF-8 文件前加入 BOM。
- 目标程序没有容忍 BOM。

提前策略：
- 使用明确支持无 BOM UTF-8 的写入方式。
- 不用不确定编码的命令直接覆盖关键配置。

验证方式：
- 写入后立即运行目标程序的配置检查命令。
- 读取文件首字节确认没有 BOM。

## 通用问题：命令行长文本或中文参数变形

场景：
- 向脚本传递中文、多行文本、JSON 或较长消息内容。

典型现象/报错：
- 中文乱码。
- JSON 解析失败。
- 参数被截断或引号转义错误。

原因判断：
- shell、脚本运行时和外部程序之间的编码或转义规则不一致。

提前策略：
- 优先用 UTF-8 临时文件、标准输入或 UTF-8 Base64 传递长文本。
- 处理路径时使用原生命令和 literal path 参数。

验证方式：
- 写入或发送前后抽查关键中文片段。
- 不只看退出码。

## 通用问题：把本地运行数据误提交

场景：
- 框架仓库和真实运行工作区共用一个目录。

典型现象/报错：
- `git status` 出现日志、缓存、会话、下载文件或私有数据。
- 模板文档中混入真实组织名称、人员标识或业务描述。

原因判断：
- `.gitignore` 不是默认拒绝模式。
- 新增目录时没有先确认是否属于可复用框架。

提前策略：
- 使用默认拒绝的 `.gitignore`。
- 只白名单放行文档、模板和脚本骨架。
- 提交前扫描路径名和文件内容。

验证方式：
- 检查 `git diff --cached --name-status`。
- 对 staged/HEAD 文件树运行敏感关键词扫描。
